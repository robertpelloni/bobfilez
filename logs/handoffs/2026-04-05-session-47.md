## Investigation performed

### 1. bobfilez GUI bootstrap check
Inspected `gui/omni/src/main.cpp` and confirmed the active native GUI path uses:
- `QGuiApplication`
- `QQmlApplicationEngine`
- `QQmlContext`
- `qmlRegisterType(...)`

### 2. BTK declarative API generation check
Inspected BTK declarative headers and confirmed they are centered on:
- `QDeclarativeEngine`
- `QDeclarativeComponent`
- `QDeclarativeView`
- `QDeclarativeItem`

### 3. Negative search for modern QQml/QQuick provider surface
Searched the BTK tree and found no real:
- `QQmlApplicationEngine`
- `QQmlEngine`
- `QQmlContext`
- `QQuickItem`
- `QQuickView`
- `qqml*.h`
- `qquick*.h`

### 4. Declarative dependency footprint quantification
Measured inside `libs/btk/src/declarative`:
- 61 QtScript-related files
- 24 files referencing `QScriptDeclarativeClass`
- 23 files using `Q_DECLARE_METATYPE(...)`

Also confirmed missing expected script-internals signs such as no discovered:
- `qscriptdeclarativeclass_p.h`
- `qscriptengine.h`

## Key finding
The BTK provider gap is now clearly three layers deep:
1. `Declarative` is not exported by the current BTK package surface
2. when experimentally enabled, the declarative module is stale/incomplete and tied to missing QtScript-era infrastructure
3. even beyond that, BTK's current declarative surface is `QDeclarative*`-era while bobfilez's active GUI bootstrap is `QQml*`-era

## Recommended next step
The next step is now a strategic framework decision, not another small build fix:
- either pursue a major upstream BTK declarative/QQml modernization effort
- or plan how bobfilez would reduce or replace its `QQmlApplicationEngine` dependence if BTK remains the chosen framework path
